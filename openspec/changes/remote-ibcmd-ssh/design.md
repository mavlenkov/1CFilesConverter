## Context

ibcmd вызывается в 9 .sh скриптах (~36 вызовов). Для серверных ИБ на MSSQL ibcmd на Linux не работает — нужен remote-вызов через SSH на Windows-хосте.

Все вызовы ibcmd следуют одному из паттернов:

| Подкоманда | Файловый аргумент | Направление | Scope v1 |
|------------|-------------------|-------------|----------|
| `config load ... "${FILE}"` | последний | отправить | да |
| `config import ... "${DIR}"` | последний | отправить | да |
| `config apply ...` | нет | — | да |
| `infobase create ... --load="${FILE}"` | внутри флага | отправить | да |
| `infobase create ... --import="${DIR}"` | внутри флага | отправить | да |
| `config save ... "${FILE}"` | последний | получить | нет (v2) |
| `config export ... "${DIR}"` | последний | получить | нет (v2) |

SSH-доступ к rigel проверен: `ssh rigel`, `scp` работают. ibcmd на rigel: `C:\Program Files\1cv8\8.3.27.1859\bin\ibcmd.exe`. Оболочка SSH на rigel — `cmd.exe`.

## Goals / Non-Goals

**Goals:**
- Прозрачно выполнять ibcmd на удалённом Windows-хосте через SSH
- Автоматически передавать файлы/директории (scp) и транслировать пути
- Поддержать загрузку (load, import), создание (create) и применение (apply)
- Сохранить обратную совместимость: без `V8_REMOTE_HOST` поведение идентично текущему

**Non-Goals:**
- Выгрузка (save, export) через remote — будет в v2
- Поддержка remote-хостов без SSH-ключей (интерактивный пароль)
- Поддержка SSH-оболочек, отличных от cmd.exe (PowerShell, bash on Windows)

## Decisions

### 1. API функции `run_ibcmd()`

**Решение:** Функция принимает локальный путь к файлу/директории для передачи и аргументы ibcmd. В локальном режиме путь подставляется как есть, в remote — заменяется на remote-путь после scp.

```bash
# Формат вызова:
run_ibcmd <local_path_or_empty> <ibcmd_args...>

# Примеры:
run_ibcmd "${V8_SRC_PATH}" infobase config load --dbms=... --db-server=... --force
run_ibcmd "${XML_PATH}" infobase config import --dbms=... --db-server=...
run_ibcmd "" infobase config apply --dbms=... --db-server=...
run_ibcmd "${V8_SRC_PATH}" infobase create --dbms=... --create-database --load=
```

Логика:
- Первый аргумент — локальный путь к файлу/директории для передачи (пустая строка = нет файла)
- Остальные аргументы — команда ibcmd
- Для `infobase create` с `--load=` или `--import=`: функция дописывает remote-путь к флагу
- Для остальных команд: путь добавляется как последний аргумент ibcmd

В локальном режиме (`V8_REMOTE_HOST` не задан):
- Путь подставляется как есть (локальный)
- ibcmd вызывается через `"${IBCMD_TOOL}"`

В remote-режиме:
- Файл/директория копируется через scp
- Путь заменяется на remote-путь
- ibcmd вызывается через `ssh "${V8_REMOTE_HOST}"`

**Альтернатива (отвергнута):** Плейсхолдер `{REMOTE_PATH}` в аргументах — сломается в локальном режиме, хрупко.

**Альтернатива (отвергнута):** Скрипт-прокси `ibcmd` на PATH — не может различить локальные пути от аргументов СУБД.

### 2. Уникальная рабочая директория на каждый вызов

**Решение:** Каждый remote-вызов `run_ibcmd` (включая `config apply` без файлов) создаёт уникальный подкаталог в `V8_REMOTE_TEMP`:

```
V8_REMOTE_TEMP\run-<PID>-<TIMESTAMP>
```

Этот подкаталог используется:
- Для копируемых файлов/директорий (scp)
- Для `--data=` (рабочая директория ibcmd): `<unique_dir>\ibcmd_data`

Очищается целиком после завершения ibcmd, после ошибки ibcmd, и после ошибки scp (если подкаталог уже создан). Это исключает коллизии при параллельных запусках.

**Альтернатива (отвергнута):** Общий `V8_REMOTE_TEMP` без подкаталога — коллизии, удаление чужих файлов.
**Альтернатива (отвергнута):** Общий `--data=` в `V8_REMOTE_TEMP\ibcmd_data` — коллизии при параллельных вызовах.

### 3. Формат путей для scp на Windows

**Решение:** scp на Windows-хост через OpenSSH принимает формат с прямыми слешами или drive letter без двоеточия. Используем формат с кавычками:

```bash
# Файл:
scp "${local_file}" "${V8_REMOTE_HOST}":"'${remote_dir}\\'"
# Директория:
scp -r "${local_dir}" "${V8_REMOTE_HOST}":"'${remote_dir}\\'"
```

Примечание: scp интерпретирует `:` как разделитель host:path, поэтому Windows-путь с drive letter (`C:\...`) передаётся в кавычках после `:`.

### 4. Выполнение команд через SSH на Windows (cmd.exe)

**Решение:** Команда формируется с учётом `cmd.exe` как оболочки SSH:
- Путь к ibcmd.exe оборачивается в двойные кавычки: `"C:\Program Files\1cv8\...\ibcmd.exe"`
- Аргументы с пробелами — в двойных кавычках
- Обратные слеши в Windows-путях не экранируются (cmd.exe не интерпретирует `\` как escape)
- Спецсимволы cmd.exe (`&`, `|`, `^`, `%`, `(`, `)`) в значениях аргументов (пароли, имена) экранируются через `^` при необходимости. В v1 предполагается, что пароли и имена не содержат спецсимволов; если понадобится — усилить экранирование.
- Вся команда передаётся в ssh как единая строка в одинарных кавычках (для bash на Linux)

Пример результирующей команды:
```bash
ssh rigel 'chcp 65001 >nul & "C:\Program Files\1cv8\8.3.27.1859\bin\ibcmd.exe" infobase config load --dbms=MSSQLServer --db-server="RIGEL\SQL2019" --db-name=TEST_CONV --db-user=sa --db-pwd=AlteySA11 --force "C:\Temp\1c_conv\run-12345-20260401\1cv8.cf"'
```

### 4. Кодировка вывода ibcmd на Windows

**Решение:** ibcmd на Windows выдаёт диагностические сообщения в OEM-кодировке (CP866). При передаче через SSH на Linux-терминал в UTF-8 текст будет нечитаемым. Функция `run_ibcmd`:
- Определяет успех/ошибку по коду возврата ibcmd, а не по тексту вывода
- Перед вызовом ibcmd выполняет `chcp 65001` для переключения консоли в UTF-8
- Полная команда: `ssh host 'chcp 65001 >nul & "ibcmd.exe" ...'`

### 6. Трансляция параметра --data=

**Решение:** При remote-выполнении `--data=` заменяется на путь внутри уникального подкаталога: `<unique_dir>\ibcmd_data`. Функция ищет `--data=*` среди аргументов и подставляет remote-путь. Это гарантирует изоляцию рабочих данных ibcmd при параллельных запусках.

### 7. Ослабление проверки локального ibcmd

**Решение:** В `init_convert_tool()` при заданном `V8_REMOTE_HOST` проверка наличия локального файла `IBCMD_TOOL` пропускается. ibcmd будет вызываться на remote-хосте.

### 8. Область изменений в скриптах

Заменяются только вызовы ibcmd в ветках серверных ИБ (`V8_IB_SERVER`):

| Файл | Что меняется |
|------|-------------|
| `common.sh` | Новая функция `run_ibcmd()`, модификация `init_convert_tool()`, `run_update_db()` |
| `conf2ib.sh` | Вызовы `config load`, `config import`, `infobase create` для серверных ИБ |
| `ext2ib.sh` | Вызовы `config load`, `config import` для серверных ИБ |

Файловые ИБ (`--db-path`) не затрагиваются — они всегда локальные.

### 9. Новые переменные окружения

| Переменная | Описание | По умолчанию |
|------------|----------|-------------|
| `V8_REMOTE_HOST` | SSH-хост для ibcmd | (пусто — локальный режим) |
| `V8_REMOTE_IBCMD` | Путь к ibcmd.exe на remote | `C:\Program Files\1cv8\<V8_VERSION>\bin\ibcmd.exe` |
| `V8_REMOTE_TEMP` | Базовая временная директория на remote | `C:\Temp\1c_conv` |

Значения по умолчанию инициализируются в `init_convert_tool()` при наличии `V8_REMOTE_HOST`. Выводятся в лог при `[INFO]`.

## Risks / Trade-offs

- **[Кодировка вывода]** → ibcmd на Windows выдаёт OEM (CP866). Митигация: `chcp 65001` перед вызовом ibcmd для переключения в UTF-8.

- **[Скорость scp для XML-директорий]** → Большие конфигурации могут содержать тысячи XML-файлов. scp -r будет медленным. Митигация: для крупных конфигураций можно упаковывать в tar/zip перед передачей (оптимизация в будущем, не в scope v1).

- **[Очистка remote temp при аварийном завершении]** → При аварии скрипта (kill, Ctrl+C) remote temp не очистится. Митигация: уникальные подкаталоги (`run-<PID>-<TS>`) минимизируют последствия; ручная очистка `V8_REMOTE_TEMP` безопасна.

- **[Пробелы в путях Windows]** → Путь к ibcmd.exe содержит пробелы. Митигация: явная стратегия quoting (двойные кавычки), проверена при ручном тестировании.

- **[Ошибки SSH/SCP]** → Сеть может быть недоступна. Митигация: `run_ibcmd` проверяет код возврата scp и ssh, при ошибке выводит диагностику и возвращает ненулевой код.

- **[cmd.exe оболочка]** → SSH на Windows может использовать разные оболочки. Митигация: дизайн ориентирован на cmd.exe (стандарт OpenSSH for Windows). PowerShell и bash не поддерживаются в v1.
