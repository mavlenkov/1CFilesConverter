## Context

Тестовая инфраструктура проекта состоит из 84 CMD-файлов: главный запускатель test.cmd, 8 скриптов подготовки (before/), 76 тестовых скриптов (tests/), 1 скрипт очистки (after/) и тестовые фикстуры (fixtures/bin/).

Тесты делятся на:
- **Файловые** (~60 тестов): конвертация между CF/XML/EDT/IB — работают с файловой ИБ и локальными инструментами
- **Серверные** (~15 тестов): используют серверную ИБ, RAC/RAS, sqlcmd
- **Специальные** (тест 127): запуск GUI-процесса Designer параллельно с конвертацией

Паттерн тестирования: каждый тест задаёт `TEST_CHECK_PATH` (ожидаемые файлы), главный запускатель проверяет их существование после выполнения.

## Goals / Non-Goals

**Goals:**
- Портировать test.cmd → test.sh с сохранением логики: подготовка → тесты → очистка → отчёт
- Портировать все 85 скриптов (before + tests + after) в bash
- Адаптировать серверные тесты под Linux: psql, ps/kill, Linux-пути
- Сохранить совместимость с тем же набором фикстур (fixtures/bin/)

**Non-Goals:**
- Переход на тестовый фреймворк (bats, shunit2) — сохраняем тот же подход
- Добавление новых тестов сверх существующих в CMD
- CI/CD интеграция (GitHub Actions)

## Decisions

### 1. Структура test.sh — прямой порт паттерна test.cmd

**Решение:** test.sh повторяет логику test.cmd: загрузка .env, создание каталогов out/, последовательный запуск before/ → tests/ → after/ с проверкой TEST_CHECK_PATH и подсчётом SUCCESS/FAILED.

**Альтернатива:** Использовать bats или shunit2. Отвергнута: сохраняем единообразие с CMD-оригиналом, минимизируем расхождения, не добавляем зависимости.

### 2. Вызов тестовых скриптов через source

**Решение:** Тестовые скрипты вызываются через `source` (аналог CMD `call`), чтобы они имели доступ к переменным фреймворка (TEST_COUNT, SCRIPTS_PATH, OUT_PATH и т.д.) и могли устанавливать TEST_CHECK_PATH и TEST_ERROR_MESSAGE.

**Альтернатива:** Вызов как подпроцессы (`./script.sh`). Отвергнута: тесты зависят от общих переменных и устанавливают результирующие переменные для фреймворка.

### 3. Адаптация серверных тестов

**Решение:**
- `sqlcmd` → `psql` (PostgreSQL по умолчанию на Linux, но сохраняем переменную V8_DB_SRV_DBMS)
- `tasklist/taskkill` → `pgrep`/`kill`
- `start /D` → запуск в фоне через `&`
- RAC/RAS-пути — использовать Linux-варианты из /opt/1cv8/
- `timeout /t N` → `sleep N`

### 4. Тест 127 (designer_running) — адаптация без GUI

**Решение:** На Linux 1cv8 может запускаться в headless-режиме. Используем `1cv8 DESIGNER ... &` для запуска в фоне, `pgrep -f` для поиска процессов, `kill` для завершения.

### 5. Порядок портирования — инкрементальный по группам

**Решение:** Портируем в порядке:
1. test.sh (фреймворк)
2. before/ (подготовка — от них зависят тесты)
3. tests/ по группам: 001 → 1XX → 2XX → 3XX → 4XX → 5XX
4. after/ (очистка)

Это позволяет тестировать частями: после портирования test.sh + before + группы 1XX уже можно запускать.

## Risks / Trade-offs

- **[PostgreSQL vs MSSQL]** → Серверные тесты в CMD используют sqlcmd (MSSQL). На Linux по умолчанию PostgreSQL. Скрипт должен поддерживать оба варианта через V8_DB_SRV_DBMS, но DROP DATABASE команда будет различаться. → Используем условную ветку по V8_DB_SRV_DBMS.
- **[Кириллические имена файлов]** → Фикстуры содержат файлы с русскими именами (ВнешняяОбработка1.epf). На Linux с UTF-8 это работает, но в скриптах нужно правильно обрабатывать кодировку. → Устанавливаем LC_ALL=C.UTF-8 в начале test.sh.
- **[Отсутствие EDT/ibcmd на тестовой машине]** → Тесты с EDT/ibcmd будут пропускаться если инструменты не установлены. → test.sh выводит SKIP для тестов с недоступными инструментами (аналог существующего поведения CMD — тесты просто падают, мы улучшаем).
