## Context

1CFilesConverter (v1.6.1) — набор скриптов для конвертации файлов 1С:Предприятие 8.3 между форматами CF, CFE, EPF/ERF, XML, EDT и информационными базами. Проект содержит 11 основных скриптов в двух реализациях (Windows CMD и Linux bash), общую библиотеку common.sh, примеры workflow-скриптов и тесты.

Текущее состояние: код стабилен, README содержит неформальное описание параметров и поведения. Формальных спецификаций нет. Это первое изменение в OpenSpec — создание спецификаций для существующего функционала без изменения кода.

## Goals / Non-Goals

**Goals:**
- Формализовать поведение каждого скрипта конвертации: входные/выходные форматы, параметры, переменные окружения
- Описать контракт common.sh: публичные функции, инициализацию, обнаружение инструментов
- Задокументировать граничные условия и поведение при ошибках
- Создать основу для автоматической верификации соответствия CMD и bash реализаций

**Non-Goals:**
- Изменение кода скриптов
- Спецификация примеров из `examples/` (они являются шаблонами, а не частью ядра)
- Спецификация watchman-интеграции (`wmscripts/`)
- Спецификация тестов (`tests/`)
- Описание внутреннего устройства инструментов 1С (Designer, ibcmd, ring)

## Decisions

### 1. Группировка спецификаций по типу конвертируемого артефакта

**Решение:** Пять спецификаций — по типу артефакта (config, extension, dataprocessor) + отдельно для EDT-валидации и common-library.

**Альтернатива:** Одна спецификация на каждый скрипт (11 файлов). Отвергнута из-за высокой степени дублирования — скрипты одной группы разделяют логику определения форматов и цепочки конвертации.

**Обоснование:** Скрипты conf2ib/conf2cf/conf2xml/conf2edt объединены общей логикой определения формата источника и цепочками промежуточных конвертаций. Группировка по артефакту естественно отражает эту связь.

### 2. Спецификация common.sh как отдельной capability

**Решение:** common.sh описывается как самостоятельная спецификация, а не как деталь реализации.

**Альтернатива:** Описать функции common.sh внутри спецификаций скриптов, которые их используют. Отвергнута, т.к. common.sh — переиспользуемая библиотека с собственным контрактом.

**Обоснование:** common.sh содержит ~25 публичных функций, используемых всеми bash-скриптами. Изменение в common.sh затрагивает все скрипты — нужна отдельная спецификация.

### 3. Формат описания цепочек конвертации

**Решение:** Каждая цепочка конвертации описывается через сценарии WHEN/THEN с указанием промежуточных шагов.

**Обоснование:** Ключевая сложность скриптов — автоматическое определение формата источника и построение цепочки промежуточных конвертаций (например, EDT → XML → CF). Сценарии делают эти цепочки явными и тестируемыми.

## Risks / Trade-offs

- **[Расхождение спецификации и реализации]** → Спецификация описывает наблюдаемое поведение существующего кода, а не идеальное. При обнаружении багов они фиксируются как отдельные change.
- **[Неполнота]** → Некоторые граничные условия могут быть пропущены. Спецификации будут дополняться по мере обнаружения.
- **[CMD vs bash различия]** → Не все различия между реализациями являются багами (например, пути в Windows vs Linux). Спецификация описывает поведение платформо-независимо, где возможно.
